# Master-Prompt für temporäre Stabilisierung der Sync-Logik / Lager_app
# Stand: 0.9.4+39 / 2026-04-29

Wir arbeiten an der fachlichen Stabilisierung, Prüfung, Dokumentation und punktuellen Weiterentwicklung der gesamten PocketBase-basierten Sync-Logik der Lager_app.

Dieser Prompt ist temporär und dient als durchgehende Arbeitsgrundlage, bis die Sync-Logik fachlich gefestigt, testbar und dokumentierbar abgeschlossen ist.

Ziel ist, den aktuellen Ist-Stand, die Architektur, die Konfliktstrategie, bereits umgesetzte Änderungen, bekannte Rest-Risiken und die nächsten sinnvollen Schritte so festzuhalten, dass die Arbeit jederzeit ohne erneute Grundanalyse fortgesetzt werden kann.

---

## Hauptziel

Die native App (Mobile/Desktop) arbeitet offline-first mit lokaler SQLite-Datenbank und synchronisiert gegen PocketBase.

Die Sync-Logik soll:

- lokale Änderungen sicher erkennen
- Remote-Änderungen gegen eine stabile Baseline prüfen
- Konflikte zuverlässig erkennen statt blind zu überschreiben
- bewusste Nutzerentscheidungen korrekt persistieren und respektieren
- Pull und Push für Dirty-Datensätze fachlich sauber behandeln
- Delete-vs-Remote-Edit korrekt als Konflikt behandeln
- UI-seitige Konfliktauflösung stabil in den Sync-Flow integrieren
- Duplicate-UUID-Race-Conditions robust behandeln
- Logging für kritische Sync-Sonderfälle nachvollziehbar halten
- dokumentierbar und releasefähig bleiben

Die Web-Version nutzt diese lokale Sync-Logik nicht in gleicher Form.

### `_extractBildName()` ← NEU

```dart
String? _extractBildName(dynamic data) {
  final raw = _asStringDynamicMap(data)['bild'];
  if (raw == null) return null;
  if (raw is List && raw.isNotEmpty) return raw.first.toString();
  if (raw is String && raw.trim().isNotEmpty) return raw.trim();
  return null;
}
```
PocketBase gibt bild nach File-Upload als List<String> zurück, nicht
als String. Diese Methode normalisiert beide Fälle. Wird nach CREATE und
UPDATE zur Bestimmung von remoteBildPfad verwendet.

## Zuletzt dokumentierter Root Cause der alten Logik

Die frühere Konfliktlogik nutzte `etag` gleichzeitig als:
1. Dirty-Marker
2. letzten bekannten Remote-Stand

Lokale Änderungen setzten `etag = null`, wodurch die Vergleichsbasis verloren ging.
Dadurch entstand in Konfliktfällen faktisch Last-Write-Wins statt belastbarer Konflikterkennung.

Die neue fachliche Basis trennt daher:
- Dirty-Zustand über `etag`
- Vergleichsbasis über `last_synced_etag`
- bewusste Nutzerentscheidung über `pending_resolution`

## Zuletzt dokumentierter Ist-Stand der produktiven Sync-Logik

Datei:
- `lib/services/pocketbase_sync_service.dart`

### Zentrale Hilfslogik
- `_isDirty(Artikel)`
- `_hasPendingResolution(Artikel)`
- `_isForceResolution(Artikel)`
- `_needsConflictBecauseMissingBase(Artikel)`
- `_hasRemoteChangedSinceLastSync(Artikel, remoteEtag)`
- `_isDuplicateUuidError(Object)`
- `_findRemoteRecordByUuid(String uuid)`
- `_markLocalAsSyncedFromRemote(Artikel, dynamic remoteRecord)`
- `_emitConflictIfPossible(Artikel lokal, Artikel remote)`

### Konfliktbasis
Konflikterkennung basiert produktiv auf:
- `lastSyncedEtag`

Nicht mehr nur auf:
- `etag`

### Zusätzliche konservative Konfliktregel
Wenn lokal Änderungen oder ein lokales Soft-Delete vorliegen, remote bereits ein passender Datensatz existiert und zugleich keine stabile Vergleichsbasis in `last_synced_etag` vorhanden ist, wird der Fall konservativ als Konflikt behandelt.

Das gilt konsistent für:
- Push-Update
- Push-Delete
- Pull bei lokal dirty + existierendem Remote-Record

Bewusste Ausnahmen über:
- `pending_resolution = force_local`
- `pending_resolution = force_merge`

bleiben erlaubt.

### Push-Verhalten
Für Pending-Artikel:
- Remote-Record wird über `uuid` gesucht
- bei lokalem Soft-Delete:
  - wenn `last_synced_etag` fehlt und keine Force-Resolution gesetzt ist:
    - Konflikt
  - wenn Remote seit `last_synced_etag` geändert wurde und keine Force-Resolution gesetzt ist:
    - Konflikt
  - sonst:
    - Remote-Delete
    - lokal `markSynced(uuid, 'deleted')`
- bei normalem Update:
  - wenn `last_synced_etag` fehlt und keine Force-Resolution gesetzt ist:
    - Konflikt
  - wenn Remote seit `last_synced_etag` geändert wurde und keine Force-Resolution gesetzt ist:
    - Konflikt
  - sonst:
    - `update()`
- wenn kein Remote vorhanden:
  - `create()`

### `remoteBildPfad` nach Push ← NEU

Nach erfolgreichem CREATE oder UPDATE wird `remoteBildPfad` über
`_extractBildName(created.data)` bzw. `_extractBildName(updated.data)`
bestimmt und via `markSynced(..., remoteBildPfad: ...)` in SQLite
persistiert.

Hintergrund: PocketBase liefert `bild` nach File-Upload als
`List<String>`. Der frühere Cast auf `String?` schlug still fehl →
`remoteBildPfad` blieb `null`.


### Duplicate-UUID-Recovery beim Create
Wenn `create()` an einer bereits vorhandenen `uuid` scheitert:
- Duplicate-Fall wird erkannt
- bestehender Remote-Record wird erneut per `uuid` gesucht
- lokaler Datensatz wird mit dessen Remote-ID / ETag als synchronisiert markiert
- dadurch scheitert derselbe Datensatz nicht bei jedem weiteren Sync erneut im Create-Pfad

### Konflikt-Snapshot-Strategie ← NEU

Der Konflikt-Callback wird ausschließlich im Push-Pfad ausgelöst —
niemals direkt im Pull-Pfad.

Ablauf:
1. Pull erkennt einen potenziellen Konflikt (lokal dirty + Remote geändert)
   → speichert den Remote-Stand via `saveRemoteConflictSnapshot()` in
     `conflict_snapshots` (SQLite v6)
   → löst keinen Callback aus
2. Push verarbeitet denselben Datensatz
   → lädt den gespeicherten Snapshot via `loadRemoteConflictSnapshot()`
   → ruft `_emitConflictIfPossible(lokal, snapshot)` genau einmal auf
   → UUID-Guard verhindert doppelten Callback im selben Sync-Zyklus

Fachlicher Effekt:
- Konflikt-Callback wird pro UUID maximal einmal pro `syncOnce()`-Lauf ausgelöst
- Remote-Bildinformationen sind im Snapshot korrekt enthalten
- Pull und Push sind fachlich sauber entkoppelt

### Logging im Duplicate-Recovery-Pfad
Für den Duplicate-UUID-Sonderfall existieren strukturierte Logs:
- `WARN`:
  - Duplicate erkannt, Recovery-Lookup startet
- `INFO`:
  - Recovery erfolgreich, lokaler Datensatz an bestehenden Remote-Record angehängt
- `ERROR`:
  - Recovery fehlgeschlagen, Fehler wird mit `error` + `stackTrace` geloggt

### Force-Resolution
`_isForceResolution()` behandelt aktuell:
- `force_local`
- `force_merge`

als bewussten Override-Push beim nächsten Sync.

### Pull-Verhalten
Für Remote-Records:
- wenn lokal `pending_resolution` gesetzt:
  - Pull skip
- wenn lokal dirty und keine stabile Vergleichsbasis in `last_synced_etag` vorhanden:
  - Konflikt
- wenn lokal dirty und Remote seit `last_synced_etag` geändert:
  - Konflikt
- wenn lokal dirty:
  - Pull skip
- sonst:
  - `upsertArtikel(..., etag: remoteEtag)`

### Remote-Löschungen beim Pull
Wenn ein lokal bekannter Datensatz remote fehlt:
- lokale Löschung erfolgt nur, wenn:
  - `remote_path` gesetzt
  - lokal weder dirty noch pending
- sonst:
  - lokale Löschung wird übersprungen

### Bild-Sync
Getrennt vom Textsync:
- `downloadMissingImages()`
- Bilddownload über PocketBase File API
- lokale Speicherung im Cache-Verzeichnis
- Verwendung von `setBildPfadByUuidSilent()`

### Request-Timeouts im Push-Pfad (B-014 / B-015)
Zwei Timeout-Konstanten:
- `_kPushRequestTimeout = Duration(seconds: 30)` — JSON-only Requests
  (getList, delete, Recovery-Lookup)
- `_kPushUploadTimeout  = Duration(seconds: 120)` — Datei-Uploads
  (create/update wenn `files` nicht leer)

Orchestrator-Timeout: `Duration(minutes: 5)` für den gesamten syncOnce()-Lauf.

Timeout-after-success Recovery (Fix C):
Wenn `create()` mit Bild einen TimeoutException wirft, kann der Server
den Record bereits erstellt haben. In diesem Fall greift dieselbe
Recovery-Logik wie bei Duplicate-UUID-Fehlern:
- Recovery-Lookup per uuid
- wenn Record gefunden: markSynced() → kein falscher Konflikt im Pull
- wenn Record nicht gefunden: echter Fehler, rethrow

- Navigation über `GlobalKey<NavigatorState>`

### Konflikt-UI
Konflikte werden geöffnet über:

### Konflikt-Hilfslogik
Datei:
- `lib/services/conflict_resolution_utils.dart`

Rolle:
- kleine isolierte Hilfslogik für Konfliktauflösungsdetails
- aktuell insbesondere Validierung einer belastbaren Remote-Baseline für `useRemote`

### Adapter
`_PocketBaseConflictAdapter implements SyncService`

Fachliche Rolle:
- Minimaladapter für die Konfliktauflösungs-UI im PocketBase-Flow

Produktiv relevant implementiert:
- `applyConflictResolution(...)`

Weitere vom `SyncService` erwartete Member sind aktuell nur als explizite Stub-Implementierungen vorhanden und werfen `UnimplementedError`, falls sie versehentlich genutzt werden.

Wichtige Einordnung:
- der Adapter ist bewusst eine kleine Übergangslösung für den ConflictResolutionScreen
- er ist kein vollwertiger allgemeiner SyncService

### Reales Verhalten der Konfliktauflösung

#### useLocal
- `_db.markForForceLocal(conflict.localVersion.uuid)`

#### useRemote
- `_db.upsertArtikel(conflict.remoteVersion, etag: remoteEtag)`
- `_db.clearPendingResolution(conflict.remoteVersion.uuid)`

`remoteEtag` wird über eine eigene Hilfslogik bestimmt:
- `requireRemoteBaselineEtag(conflict.remoteVersion)`

Zulässige Baseline-Quellen:
- `remoteVersion.etag`
- sonst `remoteVersion.lastSyncedEtag`

Nicht mehr zulässig:
- `remoteVersion.remotePath`
- leerer String als Fallback

Wenn keine belastbare Remote-Baseline vorhanden ist, wird die Auflösung mit `StateError` abgebrochen (fail-fast statt semantisch unsauberer Persistenz).

#### merge
- `_db.updateArtikel(mergedVersion)`
- `_db.markForForceMerge(mergedVersion.uuid)`

#### skip
- keine Persistenzänderung
- Konflikt soll beim nächsten Sync erneut erscheinen

- sequentielle Bearbeitung mehrerer Konflikte
- Fortschritt `(x/n)`
- Merge-Dialog mit manuell wählbaren Feldwerten
- Abschluss liefert Pop-Result:
  - `resolved`
  - `skipped`

## Sync-Invarianten

Diese Regeln möglichst nicht brechen:

- `uuid` ist stabil und geräteübergreifend eindeutig
- `remote_path` ist die PocketBase Record-ID
- `etag == null/leer` bedeutet dirty / pending
- `last_synced_etag` bleibt bei normalen lokalen Änderungen erhalten
- `pending_resolution = force_local` erlaubt bewussten lokalen Override-Push
- `pending_resolution = force_merge` erlaubt bewussten Merge-Push
- `deleted = 1` ist lokales Soft-Delete
- `setBildPfadByUuidSilent()` darf keinen normalen Datensync-Trigger auslösen
- `toPocketBaseMap()` darf keine lokalen Konflikt-Steuerfelder übertragen
- Duplicate-UUID beim Create darf nicht zu endlosen Wiederholungsfehlern führen
- fehlende Konfliktbasis bei existierendem Remote-Pendant darf nicht still optimistisch überschrieben werden
- `setBildPfadByUuidSilent()` und `clearBildInfoByUuidSilent()` dürfen
keinen normalen Datensync auslösen (nur Bild-Caching)
- `toPocketBaseMap()` muss `erstelltAm` und `aktualisiertAm` als
  UTC-ISO-Strings übertragen ← NEU
- `_extractBildName()` ist die einzige zulässige Stelle zur Normalisierung
  von PocketBase `bild` → `remoteBildPfad` ← NEU
  - `saveRemoteConflictSnapshot()` / `loadRemoteConflictSnapshot()` sind die
  einzigen zulässigen Stellen zum Lesen/Schreiben von `conflict_snapshots`
- der Konflikt-Callback darf pro UUID maximal einmal pro `syncOnce()`-Lauf
  ausgelöst werden (UUID-Guard im Push-Pfad) ← NEU

> `remoteBildPfad` (SQLite-Spaltenname: `remoteBildPfad`) wird nur durch
> `markSynced()` nach erfolgreichem Push oder durch `upsertArtikel()` nach
> erfolgreichem Pull gesetzt. Lokale manuelle Änderungen an diesem Feld
> sind nicht vorgesehen.

## Arbeitsauftrag für jede weitere Session

Wenn ich dir eine Datei, einen Fehler, einen Test oder einen Teilbereich der Sync-Logik zeige, dann möchte ich immer:

1. Prüfung, ob der gezeigte Stand fachlich zur Zielarchitektur passt
2. Benennung konkreter Abweichungen oder Risiken
3. Vorschlag der minimal-invasiven nächsten Änderung
4. Hinweis, welche Tests betroffen sind oder ergänzt werden sollten
5. kurze Bewertung des Regressionsrisikos
6. kurze Abschlussdokumentation für diesen Schritt
7. passenden Commit-Text
8. optional passenden Doku-/Changelog-Baustein

Wenn die gezeigten Informationen dafür nicht ausreichen, fordere bitte zuerst gezielt die fehlenden Dateien, Teststellen oder Doku-Ausschnitte an.
