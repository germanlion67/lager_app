## 2) App-Einstieg & Navigation (`main.dart`)

`main.dart` (~17 KB) ist der zentrale Orchestrator. Er enthält:

### Wichtige Architektur-Details

- **GlobalKey:** Nutzt `GlobalKey<NavigatorState> navigatorKey` in `main.dart`, um den `ConflictCallback` im `SyncOrchestrator` zu registrieren. Dies ermöglicht UI-Dialoge (Merge/Konflikt) auch während Navigations-Übergängen.
- **Sync-Init:** Der Sync startet erst nach erfolgreichem Login/Setup oder App-Resume. Bei Resume wird die DB explizit neu geöffnet, um `database closed` Fehler zu vermeiden (B-004).

### Initialisierungsreihenfolge (`main()`)

1. `AppConfig.init()` — Runtime-Konfiguration laden (Web: window.ENV_CONFIG)
2. `AppConfig.validateForRelease() + validateConfig()`
3. `FlutterError.onError` + `PlatformDispatcher.instance.onError` — globale Fehler
4. `platform.initDesktopDatabase()` — SQLite-FFI-Init (nur Native)
5. `PocketBaseService().initialize()` — PB-Client aufbauen
6. `AppLockService().init()` — Biometrie-Service (nur Native)
7. `runApp(MyApp())`

### Screen-Prioritätskette (`_buildHome()`)
1. ServerSetupScreen    → keine PB-URL konfiguriert
2. Lade-Spinner         → Auth-Token wird geprüft (Auto-Login)
3. LoginScreen          → nicht eingeloggt (außer PB_DEV_MODE=1)
4. AppLockScreen        → App gesperrt (Biometrie-Overlay über Haupt-App)
5. ArtikelListScreen    → Normalzustand

### Sync-Lifecycle
  - Initialer Sync nach Login/Setup via `SyncOrchestrator.runOnce()`
  - Periodischer Sync alle 15 Minuten via `Timer.periodic`
  - WLAN-Only-Sync konfigurierbar (`wifi_only_sync` in SharedPreferences)
  - Sync-Services werden nach Server-Setup neu initialisiert
  - `AppLockService.onAppPaused()` / `onAppResumed()` via `WidgetsBindingObserver`

### Dev-Mode
```bash
flutter run --dart-define=PB_DEV_MODE=1  # überspringt Login-Screen
```

### PocketBase Collections
| Collection          | Beschreibung                       | Lokales Pendant         |
| :------------------ | :--------------------------------- | :---------------------- |
| `artikel`           | Haupt-Artikeldaten                 | ✅ SQLite `artikel`     |

`attachments`-Limits:

 - Max 20 Anhänge pro Artikel (`kMaxAttachmentsPerArtikel`)
 - Max 10 MB pro Datei (`kMaxAttachmentBytes`)
 - MIME-Whitelist: `kErlaubteMimeTypes` in `attachment_model.dart`
 - Validierungsreihenfolge: Größe → Anzahl → MIME (`validateAttachment()` in `attachment_utils.dart`)
 - Auth-pflichtig seit v0.7.3 (M-009)

### Sync-Invarianten (niemals brechen)
- **uuid**        → stabil, geräteübergreifend eindeutig, nie ändern
- **remote_path** → Bindung an PocketBase Record-ID
- **etag:** Entspricht dem `updated`-Zeitstempel (ISO 8601) von PocketBase. `NULL` = lokale Änderung ausstehend ("Pending").
- **Konflikt-Erkennung:** Vor jedem `PATCH` wird der Remote-Status geprüft. Weicht `remote['updated']` vom lokalen `etag` ab, MUSS der `onConflictDetected`-Callback gefeuert werden (B-005).

deleted = 1 → Soft-Delete lokal → Hard-Delete beim nächsten Push

### Sync-Ablauf
Push (LOCAL → REMOTE):
  etag IS NULL AND deleted = 0:
    remote_path leer  → POST (create)
    remote_path gesetzt → PATCH (update)
  etag IS NULL AND deleted = 1:
    → DELETE auf PocketBase → lokal entfernen

Pull (REMOTE → LOCAL):
  updated_at > last_sync → upsert lokal
  lokal vorhanden, remote nicht → deleteArtikel()

Nach Record-Sync:
  - **downloadMissingImages():** Lädt Bilder nur, wenn lokal nicht vorhanden ODER Datei-Größe == 0 Byte (B-003 Fix).
  - **setBildPfadByUuidSilent():** Setzt Pfad ohne den `updated_at` Zeitstempel zu ändern (verhindert Sync-Loops).

### Konfliktlösung
 - Erkennung via ETag-Vergleich (`SyncService.detectConflicts()`)
 - UI: `ConflictResolutionScreen` mit Side-by-Side-Vergleich
 - Optionen: Lokal behalten / Server übernehmen / Manuell zusammenführen / Überspringen
 - Merge-Dialog: `_MergeDialog` (private, via "Manuell zusammenführen"-Button)

### Bild-Fallback-Kette (Mobile/Desktop)
| Priorität | Quelle                                  | Widget                      |
| :-------- | :-------------------------------------- | :-------------------------- |
| 1         | Lokales Thumbnail (`thumbnailPfad`)     | `_LocalThumbnail`           |
| 2         | Lokales Vollbild (`bildPfad`)           | `_LocalThumbnail` / `ArtikelDetailBild` |
| 3         | PocketBase-URL (direkt via HTTP)        | `_buildPbFallback()`        |
| 4         | Placeholder-Icon                        | `_BildPlaceholder`          |

## 6) Authentifizierung & App-Lock
### Login-Flow (M-009)
  - `LoginScreen`: E-Mail/Passwort, Validierung, Loading-State
  - Auto-Login beim Start: Token-Refresh via `PocketBaseService.refreshAuthToken()`
  - Auth-Gate in `main.dart` mit Screen-Prioritätskette (siehe Abschnitt 2)
  - Logout: `PocketBaseService.logout()` + Sync-Timer stoppen
  - Dev-Mode: `--dart-define=PB_DEV_MODE=1` überspringt Login

### App-Lock (F-001/F-002, ab v0.8.2)
  - Paket: `local_auth: ^3.0.1`
  - Service: `AppLockService` (Singleton, SharedPreferences-Persistenz)
  - Screen: `AppLockScreen` — automatischer Biometrie-Start via `addPostFrameCallback`
  - Fallback: Geräte-PIN/Pattern wenn Biometrie nicht verfügbar
  - Timeout: Konfigurierbar in Minuten (Slider im Settings-Screen, Standard: 5 Min)
  - Lifecycle: `WidgetsBindingObserver` → `onAppPaused()` / `onAppResumed()`
  - Verfügbarkeitsprüfung: `canCheckBiometrics` + `isDeviceSupported()` vor Aktivierung

## 9) Logging (AppLogService) — Pflicht

```dart 
final _log = AppLogService.logger;  // Kurzreferenz am Datei-Anfang

_log.t('trace')   // Sehr detailliert
_log.d('debug')   // Entwicklung
_log.i('info')    // Wichtige Ereignisse
_log.w('warning') // Unerwartetes, aber kein Fehler
_log.e('error', error: e, stackTrace: st)  // Fehler mit Stack
_log.f('fatal', error: e, stackTrace: st)  // Kritische Fehler
```

#### Regeln:

  - Keine neuen `print()` / `debugPrint()` einführen
  - Fehler immer mit `error`: + optional `stackTrace:`
  - `AppErrorHandler.classify()` für automatische Exception-Klassifizierung
  - 8 verbleibende `debugPrint` in `app_log_io.dart` sind absichtlich (zirkuläre Abhängigkeit)
