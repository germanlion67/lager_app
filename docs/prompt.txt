## 2) Datenbank: Besonderheiten, Schema, Sync
Persistente Speicher:
1) Lokal: SQLite (`artikel.db`)
2) Remote: PocketBase (SQLite `data.db`)

### Lokales Schema (SQLite) — Sync-relevante Mechaniken
- Globaler Identifier: `uuid` (stabil, geräteübergreifend)
- Server-Bindung: `remote_path` (PocketBase Record-ID)
- Pending-Erkennung: `etag == NULL` bedeutet lokale Änderung "noch nicht gepusht"
- Delta-Sync: `updated_at` + `sync_meta.last_sync`
- Soft-Delete: `deleted` (0/1)
- Silent-Update: `setBildPfadByUuidSilent()` setzt nur `bildPfad` ohne `updated_at`/`etag` zu ändern → kein erneuter Push

Dokumente:
- Lokale Tabelle `artikel_dokumente` mit `artikel_uuid` + `uuid/etag/updated_at/deleted/remote_path`
- Dokumente werden separat synchronisiert (eigener Zyklus)

PocketBase:
- Collections: `artikel`, `artikel_dokumente`, `attachments`
- Attachments: nur PocketBase, kein SQLite-Pendant
  - Limits: max 20 pro Artikel (`kMaxAttachmentsPerArtikel`), max 10 MB pro Datei (`kMaxAttachmentBytes`)
  - MIME-Whitelist: `kErlaubteMimeTypes` in `attachment_model.dart`
  - Validierung: `validateAttachment()` in `attachment_utils.dart` (Größe → Anzahl → MIME, in dieser Priorität)
  - API-Regeln: Auth-pflichtig seit v0.7.3 (M-009)

Sync-Regeln:
- Offline-First, Push/Pull, Delta-basiert
- Soft-Delete lokal → remote Hard-Delete beim Push
- Konflikte existieren (Pipeline + UI); nicht "wegvereinfachen", sondern integrieren
- Nach Record-Sync: `downloadMissingImages()` lädt fehlende Bilder von PocketBase File-API

## 3) Theming / UI-Regeln (Material 3)
Theming ist zentralisiert und Token-basiert:
- Keine Magic Numbers: nutze AppConfig Tokens (Spacing/Radien/IconSizes/Opacity/Layout)
- Keine `Colors.white/black` (oder andere harte Farben) – nutze `Theme.of(context).colorScheme.*`
- `ThemeMode.system` ist Standard, Dark Mode muss funktionieren
- `withOpacity()` → `withValues(alpha: AppConfig.opacity*)` (deprecated API vermeiden)
- ~560 von ~600 Hardcodes bereits migriert (~93%), ~41 bewusst beibehalten (dokumentiert in THEMING.md)

## 4) Testing-Strategie (muss bei Änderungen mitgezogen werden)
Tests: `cd app && flutter test`

Aktueller Stand (v0.8.0): **451 Tests**, 18 Dateien, 3 skipped

### Bestehende Test-Patterns respektieren:
- **DB-Integration:** `sqflite_common_ffi` In-Memory DB via `injectDatabase()` + `ArtikelDbServiceTestHelper`
- **Widget-Tests:** Timer-/Singleton-Probleme vermeiden:
  - DI via Interface (`NextcloudServiceInterface` → `NoOpNextcloudService`)
  - `SyncStatusProvider` → `FakeSyncStatusProvider` aus `test/helpers/`
  - `initialArtikel: []` überspringt async DB-Load
  - `pump(Duration)` statt `pumpAndSettle()` wenn Timer/Async sonst hängen
  - `setSurfaceSize()` + `addTearDown` bei Viewport-sensitiven Tests
- **Reine Modell/Utility-Tests:** Keine Mocks nötig, direkt testen:
  - Konstruktor, Factory-Methoden, Getter, Gleichheit, `toString()`
  - Typ-Koercion testen (String→int, double→int, null-Handling)
  - Grenzwerte testen (0, 1, exakt am Limit, knapp drüber)
  - Konstanten/Whitelists validieren (enthält erwartete Typen, keine unsicheren)
- **Mockito:** Für Service-Tests mit externen Abhängigkeiten (`MockNextcloudClient`, `MockArtikelDbService`)

### Test-Kategorien:
| Kategorie | Dateien | Beispiel |
|---|---|---|
| Unit (Modelle) | artikel_model, attachment_model, nextcloud_credentials | Konstruktor, fromMap/toMap, copyWith, Gleichheit |
| Unit (Utils) | uuid_generator, image_processing, attachment_utils | Validierung, Formatierung, MIME-Erkennung |
| Unit (Services) | backup_status, app_log_service, sync_status_provider | fromJson, State-Änderungen, Stream-Events |
| Integration (DB) | artikel_db_service | In-Memory SQLite, alle CRUD-Methoden |
| Widget | artikel_list/detail/erfassen, conflict_resolution | DI-basiert, pump statt pumpAndSettle |
| Performance | import_500_smoke | Erfordert externe Testdaten |

Wenn du neue Abstraktionen einführst (pragmatisch erlaubt):
- Bevorzuge kleine Interfaces + Constructor Injection
- Gib Test-Doubles/Mocks/NoOp-Implementierungen mit, damit Tests stabil bleiben
- Neue Test-Helper in `test/helpers/` ablegen
