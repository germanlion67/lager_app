# PROMPT (AI-Coding-Agent, pragmatisch) — lager_app (germanlion67/lager_app, Branch: main)

Du bist ein AI-Coding-Agent in diesem Repository. Arbeite repo-intern und orientiere dich primär an bestehenden Patterns.
Du DARFST pragmatisch neue Abstraktionen/Interfaces/Helper einführen, wenn es Testbarkeit, Wartbarkeit oder Plattform-Trennung verbessert.
Pflicht: Jede neue Abstraktion kurz begründen (1–3 Sätze) und minimal halten.

## 0) Kontext / Stack
- Monorepo:
  - `app/` = Flutter Hauptanwendung
  - `server/` = PocketBase Backend (inkl. `pb_migrations/`, `pb_data/`, Backups)
  - `packages/runtime_env_config/` = dynamische ENV-Injektion zur Laufzeit
- Plattformen (Hybrid-Cloud / Offline-First):
  - Mobile/Desktop: lokale SQLite + Sync zu PocketBase
  - Web: Flutter Web (SPA) greift direkt auf PocketBase zu (keine lokale SQLite)

## 1) Architektur & Struktur (wichtig für jede Änderung)
Halte dich an die vorhandene Struktur unter `app/lib/`:
- `config/`  → zentrale Konfiguration (AppConfig/AppTheme/AppImages)
- `core/`    → Plattform-Abstraktionen (z.B. Logger, Env)
- `models/`  → Datenklassen
- `services/`→ Business-Logik (API, DB, Sync, DokumentSync)
- `screens/` → UI Pages
- `widgets/` → wiederverwendbare UI-Bausteine
- `utils/`   → Helper

Plattform-spezifischer Code:
- Nutze Conditional Imports (`*_io.dart` vs `*_stub.dart`) wenn `dart:io` / Dateisystem betroffen ist.
- Web darf keine `dart:io`-Abhängigkeiten enthalten.

## 2) Datenbank: Besonderheiten, Schema, Sync
Persistente Speicher:
1) Lokal: SQLite (`artikel.db`)
2) Remote: PocketBase (SQLite `data.db`)

### Lokales Schema (SQLite) — Sync-relevante Mechaniken
- Globaler Identifier: `uuid` (stabil, geräteübergreifend)
- Server-Bindung: `remote_path` (PocketBase Record-ID)
- Pending-Erkennung: `etag == NULL` bedeutet lokale Änderung “noch nicht gepusht”
- Delta-Sync: `updated_at` + `sync_meta.last_sync`
- Soft-Delete: `deleted` (0/1)

Dokumente:
- Lokale Tabelle `artikel_dokumente` mit `artikel_uuid` + `uuid/etag/updated_at/deleted/remote_path`
- Dokumente werden separat synchronisiert (eigener Zyklus)

PocketBase:
- Collections: `artikel`, `artikel_dokumente`, `attachments` (nur PocketBase, kein SQLite-Pendant; Limits/MIME-Whitelist beachten; aktuell API offen, Auth später)

Sync-Regeln:
- Offline-First, Push/Pull, Delta-basiert
- Soft-Delete lokal → remote Hard-Delete beim Push
- Konflikte existieren (Pipeline + UI); nicht “wegvereinfachen”, sondern integrieren

## 3) Theming / UI-Regeln (Material 3)
Theming ist zentralisiert und Token-basiert:
- Keine Magic Numbers: nutze AppConfig Tokens (Spacing/Radien/IconSizes/Opacity/Layout)
- Keine `Colors.white/black` (oder andere harte Farben) – nutze `Theme.of(context).colorScheme.*`
- `ThemeMode.system` ist Standard, Dark Mode muss funktionieren

## 4) Testing-Strategie (muss bei Änderungen mitgezogen werden)
Tests: `cd app && flutter test`

Bestehende Test-Patterns respektieren:
- DB-Integration: `sqflite_common_ffi` In-Memory DB
- Testbarkeit via `injectDatabase()` / Test-Helper
- Widget-Tests: Timer-/Singleton-Probleme vermeiden (DI via Interface, NoOp-Doubles)
- Häufig `pump(Duration)` statt `pumpAndSettle()` wenn Timer/Async sonst hängen

Wenn du neue Abstraktionen einführst (pragmatisch erlaubt):
- Bevorzuge kleine Interfaces + Constructor Injection
- Gib Test-Doubles/Mocks/NoOp-Implementierungen mit, damit Tests stabil bleiben

## 5) Dokumentation aktualisieren (relevant & minimal)
Wenn deine Änderung Verhalten, Architektur, DB, Sync, Theming, Logging oder Tests beeinflusst, dann aktualisiere die passenden Doku-Dateien:

Primär:
- `docs/ARCHITECTURE.md` (Architektur/Struktur/Flows)
- `docs/DATABASE.md` (Schema, Sync-Regeln, Indizes, Limits)
- `docs/THEMING.md` (Tokens, Best Practices, neue UI-Konventionen)
- `docs/LOGGER.md` (Logging-Regeln, neue Log-Events/Viewer)
- `docs/TESTING.md` (neue/angepasste Tests, Run-Commands, neue Helpers)

Regeln:
- Doku-Änderungen sind kurz, präzise, und spiegeln die Implementierung 1:1 wider
- Wenn du neue Interfaces/Helper einführst, dokumentiere kurz “Warum” + “Wie testen”
- Wenn du neue ENV-Variablen/Deployment-relevante Dinge einführst: zusätzlich README/Setup/Deployment-Dokus anpassen, wo es hingehört

## 6) Logging (AppLogService) — Pflicht bei neuen Flows
Zentrales Logging: `AppLogService.logger`
- Verwende Level korrekt (trace/debug/info/warning/error)
- Fehler immer mit `error` + optional `stackTrace`
- Keine neuen `print()`/`debugPrint()` einführen
- Logs sind theming-integriert (Viewer), daher verständliche Messages

Wenn ich dir eine Aufgabe gebe:

5. Abschlussbericht:
